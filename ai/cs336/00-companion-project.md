# CS336 Companion Project — LabMate

> **The idea.** Just as your AWS study runs in parallel with an AWS project, this file specs out an *LLM engineering* project that runs alongside CS336. Each lecture feeds a concrete milestone. At the end you'll have a locally-deployable, domain-specialized coding assistant *and* a rehearsal of every technique the course teaches.
> **The learning contract.** Every phase forces you to use PyTorch (from learn-pytorch.io / the CS336 lectures) on a real problem, so the concepts don't stay abstract. If you finish, you have something to demo in an interview: end-to-end training + fine-tuning + eval + deployment of an open-weight model.
> **The deliverable.** A ≤ 3 B-parameter open-weight code model, LoRA-fine-tuned on scientific-Python patterns (with an optional Melio-specific pass), 4-bit-quantized, running locally on your laptop via `llama.cpp` or `vllm`, with a small CLI + optional VS Code sidebar.

---

## 1. The pitch, in one line

**LabMate** — a Qwen 2.5-Coder 3B model continually pretrained on scientific Python code (numpy / pandas / scikit-image / opencv / biopython / matplotlib / scipy) and instruction-fine-tuned to write image-analysis and data-processing scripts the way a working bioimage-analysis engineer would.

Swap `scientific Python` → `Melio's actual codebase` and you get **MeltPilot**, a Melio-internal variant. Same infrastructure, just a different training corpus.

## 2. Why this project earns its keep

- **Uses the whole CS336 stack.** Tokenization (L1), PyTorch + resource accounting (L2), architecture (L3), MoE optional (L4), kernels & Triton (L5–6), parallelism (L7–8), scaling laws (L9–11), inference (L10), evaluation (L12), data curation (L13–14), alignment SFT/DPO (L15–17). Every lecture becomes a concrete decision you make in this project.
- **Bounded scope.** A 3 B base + LoRA fits on a single 24 GB consumer GPU (or a 40 GB A100 on Colab Pro / Lambda / Runpod for ~$0.60/hr). No frontier-cluster fantasies.
- **Industry-relevant.** Domain-specialized code assistants are exactly what teams at every hardware company, life-sciences startup, and internal-tooling group build in-house today. It maps cleanly onto real job descriptions.
- **Interview-portfolio worthy.** You'll be able to say: "I pretrained on X tokens from Y sources, LoRA-fine-tuned Z with these hyperparams, evaluated on HumanEval + a custom bio-Python benchmark, quantized to 4-bit, and deployed locally. Here's the loss curve. Here's the eval numbers. Here's a demo." That's exactly the story ML-engineering interviewers want.

## 3. Base model — why Qwen 2.5-Coder 3B

| Model | Params | Context | License | Vocab | Notes |
| --- | --- | --- | --- | --- | --- |
| **Qwen 2.5-Coder 3B** | 3.09 B | 32k | Apache 2.0 | 151,646 | **Recommended.** Best small-model code performance in its class; permissive license. |
| Llama 3.2 3B | 3.2 B | 128k | Llama 3 CL | 128,256 | Broader corpus, weaker on code. |
| Phi-3.5-mini (3.8 B) | 3.8 B | 128k | MIT | 32,064 | Very compact vocab; strong reasoning. |
| DeepSeek-Coder-V2-Lite 16B | 15.7 B (2.4 B active MoE) | 128k | DeepSeek | 100,000 | Best raw code — but 16 B parameters need more VRAM even at 4-bit. |

Pick **Qwen 2.5-Coder 3B** for the main track. It fits on a laptop after quantization, its tokenizer is CJK-strong (useful even if you only care about English — the tokenizer is just better at code and long tokens), and Apache 2.0 means you can ship it internally at Melio if you want.

## 4. Data — the highest-leverage decision

The base model has already seen mainstream Python. Your value-add is *scientific-domain* patterns and, optionally, *Melio-house* patterns.

### Track A — Public scientific-Python corpus (build this first)

Sources to scrape and clean:
- **GitHub repos** matching the tags `biomedical`, `bioimage-analysis`, `image-processing`, `microscopy`, `qpcr`, `scikit-image`. Filter permissive licenses (MIT / Apache / BSD). Aim for ~1–5 GB of Python source.
- **Jupyter notebooks** from `nb-lab` / `Papermill` example collections, and from `scikit-image`, `napari`, `cellpose`, `stardist`, `deepcell` documentation. Convert `.ipynb → .py`.
- **Docstring-rich documentation** from the top scientific-Python libraries (numpy, scipy, pandas, matplotlib, scikit-image, opencv-python, biopython, plotly, seaborn, statsmodels).
- **Method sections + code appendices from bio-open-access papers** (PubMed Central OA subset — legally clean for research).

Post-processing:
- Dedup (MinHash — CS336 Lecture 13).
- Filter by heuristics: minimum lines, presence of `import numpy` / `import cv2` / `import skimage`, no auto-generated code (`# Generated by …`).
- Tokenize once, memory-map (§L2 in these notes).

Target size: **200 M – 1 B tokens.** That's ~0.7–3 GB pre-tokenized in uint32.

### Track B — Melio-house corpus (optional second pass, when you have Track A working)

Same pipeline, but on:
- Your own analysis scripts (`sarthak-analysis-scripts`, `Melio_Packages`, `meltgen`, `meliods`).
- Pipeline documentation.
- Any internal notebook history — sanitized for secrets.

This is where LabMate becomes *your team's* assistant, and where the interview story gets more compelling.

## 5. The phased milestone map

Each phase is scoped to be finishable in 1–3 evenings, and each depends on material from a specific CS336 lecture. You can start Phase 0 today; you don't need to be through the whole course to begin.

### Phase 0 — Set up the ground floor  *(before or during Lecture 1)*

- **Repo scaffolding.** `pyproject.toml`, `uv` or `poetry`, `pre-commit` with `ruff`. This is portfolio hygiene.
- **Hardware baseline.** Verify you can run Qwen 2.5-Coder 3B inference at 4-bit locally. On a Mac: `ollama run qwen2.5-coder:3b`. On Linux/Windows with GPU: `llama.cpp` + a Q4_K_M GGUF from Hugging Face.
- **Deliverable:** a working local inference of the un-modified base model, plus a short benchmark note on tokens/sec.

### Phase 1 — Tokenizer analysis  *(applies Lecture 1)*

- Take your Track A corpus (or a subset). Compute the compression ratio (bytes/token) of Qwen 2.5-Coder's tokenizer on it.
- Compare against Llama 3's tokenizer and GPT-4o's `o200k_base` on the same corpus.
- **Deliverable:** a `notebooks/01_tokenizer_analysis.ipynb` with a table + one bar chart. The chart will show *how much of your context Qwen wastes on scientific-Python patterns vs. mainstream Python*. This is a portfolio-quality artifact and takes 2 hours.
- **Interview talking point:** "I ran a compression-ratio study on our target corpus before choosing a base model, and here's what I learned."

### Phase 2 — Reproduce the 6ND arithmetic on your target run  *(applies Lecture 2)*

- Pick a hypothesis: "I will continually pretrain Qwen 2.5-Coder 3B on 500 M tokens with a LoRA rank of 16 on all attention + MLP projections."
- Compute: parameter count (base + adapters), FLOPs (6 × N_train × D), memory footprint (params fp16 + LoRA fp32 + gradients + Adam state + activations).
- Predict wall-clock at your target hardware's known throughput. Then run a 10-min mini-experiment to *measure* MFU and confirm the prediction is within 30%.
- **Deliverable:** `docs/resource_budget.md` — the same napkin math Percy did in Lecture 2, applied to *your* run. Save with a "predicted vs actual" table.

### Phase 3 — Continual pretraining  *(applies Lectures 3, 6, 7, 9)*

- Load Qwen 2.5-Coder 3B in bf16.
- Attach LoRA adapters (rank 16, α = 32, target `q_proj / k_proj / v_proj / o_proj / gate_proj / up_proj / down_proj` — the Llama-family standard) via `peft`.
- Train on Track A corpus for ~500 M tokens with a cosine schedule + 3% warmup, AdamW (β₁ = 0.9, β₂ = 0.95, weight decay 0.1), LR = 1e-4.
- Log every step: loss, tokens/sec, MFU, memory watermark, wall-clock. Use `wandb` for the plots — recruiters like the visual.
- **Deliverable:** a saved checkpoint `checkpoints/labmate-pretrain-v1/`. A `training_curve.png` for the portfolio.

### Phase 4 — Instruction dataset  *(applies Lecture 15)*

Build a small SFT dataset — a few thousand `(prompt, response)` pairs — for scientific-Python tasks. Two ways:
- **Distillation.** Send natural-language task descriptions ("extract the largest connected region from this binary mask using scikit-image") to a stronger model (Claude Sonnet, GPT-4o, DeepSeek-V3 — whichever you have access to), get back polished response code, curate.
- **From docstrings.** Mine your Track A corpus: for every well-documented function, turn the docstring into a prompt and the function body into a response.

Target: 5,000–20,000 high-quality pairs. Curated > large, always.

**Deliverable:** `data/labmate-sft.jsonl`, spot-audited so bad examples don't poison the fine-tune.

### Phase 5 — SFT  *(applies Lecture 15)*

Fine-tune LabMate-Pretrain-v1 on `labmate-sft.jsonl` for 2–3 epochs. Same optimizer profile, LR = 5e-5, batch = whatever fits, use `SFTTrainer` from TRL or a hand-rolled loop.

**Deliverable:** `checkpoints/labmate-sft-v1/`. A qualitative comparison of ~20 prompts against the base model, side-by-side.

### Phase 6 — Preference tuning  *(applies Lecture 16, optional)*

If you have time and interest: generate two completions per prompt for a subset, rank them by hand (or with a stronger model as judge), then train with **DPO** (Direct Preference Optimization — much simpler than PPO and the current small-project default). This gives you the *alignment* half of your interview story.

### Phase 7 — Evaluation  *(applies Lecture 12)*

Assemble a small evaluation harness — don't skip this, it's what separates a demo from a project.
- **HumanEval + MBPP** — standard Python benchmarks. Report pass@1.
- **A custom benchmark of 30 hand-written bio-Python tasks.** You author these. Include an oracle answer + a checker (either automatic tests or a rubric).
- **Perplexity on a held-out slice of Track A.** Report bits-per-byte for cross-tokenizer honesty (§L1).

Compare: base model → pretrained → SFT'd → DPO'd (if you did it). Chart the improvement per stage.

**Deliverable:** `eval/results.md` with a leaderboard-style table.

### Phase 8 — Quantization + local deployment  *(applies Lecture 10)*

- Convert the final model to GGUF with `llama.cpp` — Q4_K_M for the best size/quality tradeoff, Q8_0 if you want to preserve maximum quality.
- Measure tokens/sec on your laptop's CPU and GPU (Metal on Mac, CUDA on Linux/Windows).
- Compare against the base model at the same quantization: is your LabMate variant faster/slower on the same prompts? (It shouldn't be — same architecture — but it's worth confirming.)
- **Deliverable:** `labmate-3b-q4_k_m.gguf` on Hugging Face Hub (if licensing allows) + a `README.md` describing training, eval, and usage.

### Phase 9 — CLI + optional VS Code plugin  *(polish)*

- Simple Python CLI: `labmate "add a function to compute the mean signal per well from a numpy array of shape (n_wells, timesteps)"` → prints a code block.
- If you want to go further: a VS Code extension that calls a local vLLM/llama.cpp server. This is a weekend of TypeScript, not deep ML work, but it makes the demo landable.
- **Deliverable:** a 30-second screen capture for the portfolio README.

## 8. The Assignment 1 relationship

CS336 Assignment 1 asks you to build a Transformer *from scratch*, train a tokenizer, wire up AdamW, and train a tiny model on TinyStories / OpenWebText. **Do Assignment 1.** It's the "learn every atom" prerequisite for LabMate.

Then LabMate is the "now apply it to something industry-shaped" project. Assignment 1 teaches you *what* every component is; LabMate teaches you what happens when the components meet a domain, a real deployment target, and a wall-clock deadline. Both are needed for a strong interview story.

## 9. Alternative shape — Nano-CS336 (from-scratch pretraining)

If you'd rather train a model *from scratch* (mirroring the course purely) instead of fine-tuning:

- Base: a 100–300 M parameter Llama-style Transformer of your own construction (the Assignment 1 output, scaled up).
- Corpus: a well-curated blend of Common Crawl English + Python code + PubMed abstracts, tokenized with your own trained BPE.
- Training: ~5–10 B tokens.
- Cost: ~$300–$800 on rented H100s.
- Deliverable: a model that's genuinely bad at everything but useful for showing you built the whole stack.

**Recommendation: do LabMate as primary and treat Nano-CS336 as a stretch goal.** LabMate ships something usable; Nano-CS336 mostly ships a certificate that you did it. Employers want both, but usable-thing first.

## 10. Minimum viable milestones (if you're time-constrained)

If you can only do part of this, in priority order:

1. **Phase 1** (tokenizer analysis) — 2 evenings, high signal, portfolio-worthy alone.
2. **Phase 2** (resource-budget doc) — 1 evening. Shows you can plan.
3. **Phase 3** (continual pretrain) + **Phase 5** (SFT) — the training core. 1–2 weeks of chunked evenings.
4. **Phase 7** (eval) — non-negotiable. Without eval, none of the above matters for an interview.
5. **Phase 8** (quantize + deploy locally) — 1 evening; huge demo value.

Phases 4 (data curation), 6 (DPO), and 9 (CLI/VS Code) are optional polish. Skip them under time pressure.
